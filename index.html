第一章 ROS 简介

Introduction to Robot Operating System

1.1 什么是 ROS 核心概念

  • ROS (Robot Operating System) — 适用于机器人编程的框架/中间件
  • 并非 Windows/Mac 那样的传统操作系统，而是连接操作系统与 ROS 应用程序的运行时环境
  • 运行在 Linux (Ubuntu) 之上的中间件，提供通信架构
  • 为原本松散的零部件提供统一的软件平台
💡 关键词：框架、中间件、操作系统、运行时环境 — 都可以用来描述 ROS

1.2 ROS 特点

  • 分布式 · 点对点：机器人的进程可以分别运行，便于模块化修改和定制，提高容错能力
  • 多种语言支持：C++、Python、Lisp、C#、Java 等。采用语言中立的接口定义语言实现消息传送
  • 开源社区：庞大的 ROS WIKI (wiki.ros.org)，软件包达数千万个，遵从 BSD 协议，完全免费

1.3 ROS 优缺点

优点缺点
提供框架、工具和功能通信实时性能有限
方便移植系统稳定性尚不满足工业级要求
庞大的用户群体安全性上没有防护措施
免费开源仅支持 Linux (Ubuntu)
⚠️ ROS 更适合科研和开源用户，工业场景（如无人驾驶）需要做优化和定制。ROS 2.0 正在开发中。

1.4 ROS 发展历程

  • 起源于斯坦福大学人工智能实验室
  • ROS 版本按字母表顺序命名：Indigo → Jade → Kinetic → Lunar → Melodic...
  • ROS Kinetic 最佳适配 Ubuntu 16.04
  • ROS 不是 RTOS（实时操作系统），不适合 ms 级响应的任务

1.5 安装与测试 ROS

  • 安装 ROS-Academy-for-Beginners 教学包
  • 安装 RoboWare Studio 开发环境
  • 二进制包安装命令示例：sudo apt-get install ros-kinetic-packagename
🟢 测试 ROS 经典步骤（小海龟测试）： 
# 终端1: 启动 ROS Master
$ roscore

# 终端2: 启动小海龟节点
$ rosrun turtlesim turtlesim_node

# 终端3: 启动键盘控制节点
$ rosrun turtlesim turtle_teleop_key

第二章 ROS 文件系统

ROS File System & Catkin Build System

2.1 Catkin 编译系统 核心

  • ROS 对 CMake 进行了扩展 → Catkin 编译系统
  • 早期 ROS 编译系统是 rosbuild，已被逐步淘汰
  • Groovy 版本后，Catkin 正式投入使用
  • 优点：操作简化、工作效率高、可移植性好、支持交叉编译
  • ROS2 使用 Ament 编译系统
💡 演进路线：gcc/g++ → Makefile → CMake → Catkin（ROS 1.0）/ Ament（ROS 2.0）

2.2 Catkin 工作空间结构

catkin_ws/           # 工作空间根目录
├── src/             # 源代码目录（package 放这里）
├── build/           # 编译中间文件
└── devel/           # 编译生成的可执行文件
  • 创建命令：mkdir -p ~/catkin_ws/src + catkin_make
  • 编译命令：catkin_make（在工作空间根目录执行）
  • 可执行文件默认生成在 devel/ 路径
  • Package 放在 src/ 目录下

2.3 Package 软件包结构

  • 必须包含两个文件：CMakeLists.txt 和 package.xml
  • CMakeLists.txt：编译规则文件
  • package.xml：包信息描述文件
your_package/
├── CMakeLists.txt    # 必须！编译配置
├── package.xml       # 必须！包信息
├── src/              # C++ 源码
├── include/          # 头文件
├── script/           # Python 脚本
├── launch/           # launch 文件
├── msg/              # 自定义消息
├── srv/              # 自定义服务
├── action/           # 自定义动作
├── param/            # yaml 参数文件
└── urdf/             # 机器人模型文件

2.4 CMakeLists.txt 关键指令

  • add_executable() — 添加可执行文件
  • target_link_libraries() — 将库文件链接到目标文件
  • include_directories() — 引入头文件的搜索路径
⚠️ Catkin 独有的 CMake 扩展指令：add_action_files()、add_message_files()、add_service_files()、generate_messages()

2.5 Metapackage 软件元包

  • 前身：Stack（功能包集），Hydro 版本后改用 Metapackage
  • 作用：将功能接近、相互依赖的软件包放到一起
  • CMakeLists.txt 中需加入 catkin_metapackage() 宏
  • package.xml 中用 <metapackage/> 标签声明
Metapackage描述
navigation导航相关的功能包集
moveit运动规划（机械臂）功能包集
image_pipeline图像获取、处理功能包集
vision_opencvROS 与 OpenCV 交互功能包集
turtlebotTurtlebot 机器人相关功能包集

2.6 其他常见文件类型

  • .launch / .xml：ROS 程序启动文件（通常放 launch/ 路径）
  • .msg / .srv / .action：自定义消息/服务/动作文件
  • .urdf / .xacro：机器人模型描述文件
  • .yaml：参数配置文件（通常放 param/ 路径）
  • .dae / .stl：3D 模型文件（可从 SolidWorks 导出）
  • .rviz：RViz 窗口配置文件

2.7 常用命令

roscd pkg_name    # 跳转到软件包路径
rosls pkg_name    # 列出软件包内容

第三章 ROS 通信架构（一）

Communication Architecture I — Node, Master, Launch, Topic, Message

3.1 Node（节点）核心概念

  • Node 是 ROS 中最小的进程单元
  • 可执行文件运行后成为进程(process)，在 ROS 中叫做节点(node)
  • 通常一个 node 负责机器人某一个单独的功能（模块化分工）
  • 采用分布式方式，把不同功能分配到不同 node
💡 举例：底盘控制 node、摄像头驱动 node、激光雷达 node、路径规划 node — 各司其职

3.2 Master（节点管理器）

  • Master 是整个网络通信架构的管理中心
  • Node 首先在 Master 处进行注册
  • Node 之间的通信由 Master 进行"牵线"，然后点对点通信
  • 启动 ROS 的第一步：启动 Master → $ roscore
  • roscore 同时启动 rosout（日志输出）和 parameter server（参数服务器）
⚠️ 重要：Node 必须先于 ROS Master 启动？❌ 错误！必须先启动 Master，Node 才能注册和通信。

3.3 启动命令

roscore                      # 启动 Master + rosout + parameter server
rosrun pkg_name node_name     # 启动单个节点
rosrun [--prefix cmd] [--debug] pkg_name node_name [ARGS]  # 详细用法

# rosnode 常用命令
rosnode list                  # 查看所有运行中的节点
rosnode info node_name        # 查看节点信息
图3-1：Master 与 Node 之间的注册与通信关系
📖 图片说明：这张图展示了 ROS 通信架构的启动流程。首先通过 roscore 启动 Master（节点管理器），同时启动 rosout（日志节点）和 Parameter Server（参数服务器）。然后各 Node 启动时向 Master 注册，Master 负责"牵线搭桥"让 Node 之间建立点对点的直接通信。图中还展示了 Node 之间的消息发布/订阅关系——Node 间通过 Topic 进行数据交换。

3.4 Launch 文件

  • 作用：一次性启动 Master 和多个 node，减少重复操作
  • 命令：roslaunch pkg_name file_name.launch
  • roslaunch 会自动检测 roscore 是否运行，未运行则自动启动
  • Launch 文件遵循 XML 格式规范
标签作用
<launch>根标签
<node>需要启动的 node 及其参数
<include>包含其他 launch 文件
<machine>指定运行的机器
<param>定义参数到参数服务器
<rosparam>启动 yaml 文件参数
<arg>定义变量
<remap>设定参数映射
<group>设定命名空间
💡 最小 launch 示例：<launch><node name="talker" pkg="rospy_tutorials" type="talker" /></launch>

3.5 Topic（话题通信）重点

  • Topic 是 ROS 中最常用的通信方式
  • 点对点、单向、异步通信
  • Publisher（发布者）→ Topic → Subscriber（订阅者）
  • 双方先在 Master 注册 → Publisher 发布 topic → Subscriber 订阅 topic
  • Subscriber 通过回调函数 (Callback) 处理接收到的消息
  • 一个 topic 可有多个 publisher 和多个 subscriber
🟢 Topic 三个要点：
1️⃣ 异步通信 — publish() 发送后立即返回，不等反馈
2️⃣ Subscriber 通过回调函数处理消息
3️⃣ 多对多关系 — 多个发布者和多个订阅者同时存在
图3-2：Topic 通信机制 — Publish-Subscribe 异步通信模型
📖 图片说明：这是 ROS 最核心的 Topic 通信流程示意图。图中展示了完整的通信过程：① Publisher（发布者）向 Master 注册并声明自己将发布的 Topic；② Subscriber（订阅者）向 Master 注册并声明自己想要订阅的 Topic；③ Master 匹配双方后，Publisher 与 Subscriber 建立点对点直连，开始数据传输。注意整个过程是单向异步的 — Publisher 发布消息后立即返回，不等待 Subscriber 的反馈。

3.6 Message（消息格式）

  • Message = Topic 内容的数据类型/格式标准
  • msg 文件相当于"类的定义"，每次发布的内容相当于"对象"
  • 基本类型：bool, int8/16/32/64, uint, float32/64, string, time, duration, header, array[]
  • 常用 sensor_msgs/Image 包含：header, height, width, encoding, data 等字段
💡 rosmsg 命令：rosmsg list 查看所有消息类型，rosmsg show type_name 查看具体格式

3.7 常用命令速查

rostopic list              # 查看当前活跃的 Topic
rostopic echo /topic_name  # 查看 Topic 发布的内容
rostopic info /topic_name  # 查看 Topic 详细信息
图3-3：分布式 Topic 通信 — 多个节点可以同时订阅同一个 Topic
📖 图片说明：此图以摄像头画面发布与处理为例展示了 Topic 的分布式特性。摄像头节点（node1）作为 Publisher 发布 /camera_rgb 话题；图像处理节点（node2）订阅该话题进行处理；用户还可以在另一台电脑上启动图像显示节点来订阅同一话题查看画面。核心要点：一个 Topic 可以被多个节点同时发布，也可以被多个节点同时接收——体现了分布式系统的扩展性好、软件复用率高的优势。

第四章 ROS 通信架构（二）

Communication Architecture II — Service, Srv, Parameter Server, Actionlib

4.1 Service（服务通信）重点

  • 双向同步通信（请求-应答模式 Request-Reply）
  • Client 发送 Request → Server 处理后返回 Reply
  • Client 等待期间处于阻塞状态（同步通信）
  • 适用于偶尔调用的功能/具体任务（如开关传感器、拍照、逆解计算）
  • 一个 Service 只能由一个 Server 提供（多对一关系）

4.2 Topic VS Service 对比 必考

特性TopicService
通信方式异步通信同步通信
实现原理TCP/IPTCP/IP
通信模型Publish-SubscribeRequest-Reply
映射关系多对多多对一
特点回调(Callback)处理RPC 远程过程调用
应用场景连续、高频数据发布偶尔使用的功能/具体任务
举例激光雷达、里程计开关传感器、拍照、逆解计算

4.3 Srv（服务格式）

  • srv 文件描述服务的数据类型，两段式（request + reply，中间用 --- 隔开）
  • srv 可以嵌套 msg，但不能嵌套 srv
# DetectHuman.srv 示例
bool start_detect        # 请求部分
---
my_pkg/HumanPose[] pose_data   # 应答部分
💡 常用命令：rosservice type /service_name 查看类型，rossrv show type_name 查看具体格式

4.4 Parameter Server（参数服务器）

  • 特殊的"通信方式" — 节点存储参数的地方，全局共享
  • 维护着一个数据字典（键值对 key-value），每个 key 唯一
  • 更加静态，适合不常变化的参数和配置
  • 三种维护方式：① 命令行 ② launch 文件内读写 ③ node 源码
⚠️ 添加 param 不能通过 rosnode 命令，应该用 rosparam set 命令！
图4-2：参数服务器 — 键值对 (Key-Value) 字典结构
📖 图片说明：参数服务器维护着一个数据字典，每个 Key（键）都是唯一的，对应一个 Value（值）。就像查字典一样——Key 如同"部首"，唯一的 Key 对应唯一的 Value。参数服务器更加静态，适合存储不常变化的参数和配置。维护方式有三种：① 命令行（rosparam set）② launch 文件内读写 ③ node 源码中通过 API 操作。

4.5 Actionlib（动作库）重点

  • 弥补 Service 不足：适用于执行长时间任务
  • Client-Server 模式，双向通信
  • 通信过程可随时查看进度、终止请求
  • Client 可发送：目标指令(goal)、取消指令(cancel)
  • Server 可反馈：实时状态、结果信息、反馈信息
🟢 Action 文件格式（三段式）： 
# Goal (目标)
uint32 dishwasher_id
---
# Result (结果)
uint32 total_dishes_cleaned
---
# Feedback (反馈)
float32 percent_complete

4.6 四种通信方式总结

通信方式特点适用场景
Topic单向、异步、多对多连续高频数据（传感器）
Service双向、同步、多对一偶尔调用的任务（逆解）
Parameter Server静态、全局共享字典参数配置
Actionlib双向、可反馈/可取消长时间任务（导航、抓取）
图4-3：Actionlib 通信原理 — Client-Server 双向通信模式
📖 图片说明：Actionlib 采用 Client-Server 模式，弥补了 Service 在长时间任务中的不足。图中展示了通信流程：① Client 向 Server 发送目标指令 (Goal)；② Server 在处理过程中持续向 Client 发送实时反馈 (Feedback)（如完成百分比）；③ 任务完成后 Server 发送结果 (Result)；④ Client 可随时发送取消指令 (Cancel) 中止任务。这使得 Actionlib 特别适合导航、抓取等需要实时监控进度的长时任务。

第六章 roscpp — C++ 客户端库

ROS C++ Client Library

6.1 Client Library 概念 核心

  • 客户端库 (Client Library) = ROS 提供给不同语言的编程接口
  • roscpp = C++ 语言的 ROS 接口
  • rospy = Python 语言的 ROS 接口
  • 直接调用库提供的函数，即可实现 Topic、Service 等通信功能

6.2 NodeHandle

  • C++ 中使用 ros::NodeHandle 来管理节点
  • 通过 NodeHandle 创建 Publisher、Subscriber 等
  • NodeHandle 是 roscpp 的核心类

6.3 回调函数与 spin() 方法 重点

  • 回调函数 (Callback)：提前定义的处理函数，当有消息到达时触发
  • Subscriber 把消息放入队列中，spin 函数处理队首消息
spin 方法阻塞线程
ros::spin()阻塞单线程
ros::spinOnce()非阻塞单线程
ros::MultiThreadedSpin()阻塞多线程
ros::AsyncMultiThreadedSpin()非阻塞多线程
💡 最常用的是 ros::spin()（阻塞单线程）和 ros::spinOnce()（非阻塞单线程）

6.4 roscpp 主要功能

  • 节点初始化、关闭与 NodeHandle
  • Topic in roscpp — Publisher / Subscriber
  • Service in roscpp — ServiceServer / ServiceClient
  • Param in roscpp — getParam / setParam
  • 时钟 (Time) 和日志 (Log) 功能
图6-1：回调函数 (Callback) 与 spin 消息处理机制 — 消息队列的工作原理
📖 图片说明：这张图展示了 roscpp 中消息处理的核心机制。Subscriber 接收到消息后，首先将消息放入一个队列 (Queue) 中（队列长度在构建 Subscriber 时设定）。当调用 spin() 或 spinOnce() 时，程序从队首取出消息并执行回调函数 (Callback)——即提前定义好的处理函数。spin() 是阻塞式的（一直循环处理），spinOnce() 是非阻塞的（处理完队首就返回）。理解此机制是编写 ROS 节点的关键。

第七章 rospy — Python 客户端库

ROS Python Client Library

7.1 rospy 简介

第八章 TF 与 URDF

Coordinate Transforms & Unified Robot Description Format

8.1 TF (TransForm) 坐标变换 核心概念

  • TF = 坐标变换（在不同坐标系之间转换点、矢量等）
  • 本质是树状的数据结构 → tf tree
  • tf 也是一个 Topic：/tf，消息保存了 tf tree 的数据结构
  • tf 也是一个 Package，包含可视化、调试等工具
💡 核心作用：维护机器人各部件之间的坐标转换关系。例如：物体相对于头部传感器的坐标，需要转换为相对于手臂的坐标才能执行抓取。

8.2 TF 树结构

  • 每个 link（连杆）对应一个 frame（坐标系）
  • 每两个 frame 之间有一个 broadcaster（Node 发布者）维护连通
  • tf tree 不能有任何断层！两个 frame 之间必须联通
  • 如果缺少 Node 发布消息维护，两个 frame 之间连接会断开 → 系统报错
图8-1：PR2 机器人模型中的众多 frame（坐标系）— 每个 link 对应一个 frame
📖 图片说明：这张图展示了 PR2 机器人上密密麻麻的坐标系 (frame)。机器人的每个部件 (link) 上都绑定着一个 frame，例如头部摄像头有一个 frame，手臂关节有各自的 frame。当机器人"眼睛"（头部传感器）看到物体的坐标，机械臂要抓取时需要知道物体相对于手臂的坐标——这就需要 TF (坐标变换) 在不同 frame 之间进行转换。TF 维护着所有 frame 之间的树状关系。
图8-2：robot_sim_demo 运行时的 TF Tree 结构 — 树状坐标系关系
📖 图片说明：这是 robot_sim_demo 仿真运行时的实际 TF 树。每个圆圈代表一个 frame（坐标系），对应机器人上的一个 link。箭头表示坐标系之间的父子关系。关键要点：① 任意两个 frame 之间必须联通——TF 树不能有断层，否则系统报错；② 每两个 frame 之间都有一个 broadcaster（本质是一个 Publisher Node）持续发布坐标变换消息 (TransformStamped.msg) 来维持连通；③ 如果 broadcaster 停止发布，两个 frame 之间的连接就会断开。

8.3 TransformStamped.msg 消息格式

std_msgs/Header header
    uint32 seq
    time stamp
    string frame_id          # 父坐标系名称
string child_frame_id        # 子坐标系名称
geometry_msgs/Transform transform
    geometry_msgs/Vector3 translation   # 平移 (x, y, z)
    geometry_msgs/Quaternion rotation   # 旋转 (x, y, z, w) 四元数

8.4 URDF 统一机器人描述格式 重点

  • URDF (Unified Robot Description Format) — 使用 XML 格式描述机器人
  • 两个 link 之间需要 joint 来连接
  • 用 <origin> 描述相对位置和朝向关系
  • 用 <geometry> 描述形状，<color> 描述颜色
  • 显示 URDF 需要 joint_state_publisher 和 robot_state_publisher 两个节点
💡 检查 URDF 语法：sudo apt-get install liburdfdom-tools → check_urdf my_robot.urdf
图8-3：URDF 中 link（连杆）与 joint（关节）的关系 — base_link 与 right_wheel 通过 joint 连接
📖 图片说明：URDF 用 XML 格式描述机器人模型。图中以小车为例展示了核心概念：link（连杆）是机器人的刚性部件（如车身 base_link、右轮 right_wheel），joint（关节）连接两个 link 并定义它们之间的运动关系（如旋转、平移）。每个 joint 通过 <origin> 标签定义相对位置和朝向，通过 <geometry> 定义形状，通过 <color> 定义外观颜色。
图8-4：URDF 小车模型在 RViz 中的完整显示效果
📖 图片说明：这是 URDF 模型制作的最终效果——在 RViz 中显示出完整的小车三维模型。要显示 URDF 模型，需要启动 joint_state_publisher（发布关节状态）和 robot_state_publisher（发布机器人整体状态）两个节点，并配合 launch 文件导入 URDF 描述。图中可见小车的车身（base_link）、左右轮以及各 joint 的连接关系，所有 link 已经设置了正确的相对位置、形状和颜色。

附录：三维空间刚体运动

  • 旋转矩阵：9 个量描述 3 自由度旋转，具有冗余性。Pw = R·Pc + t
  • 欧拉角：3 个量 (Roll-Pitch-Yaw)，但存在万向锁问题（pitch=90° 时丢失自由度）
  • 四元数：ROS 中使用的旋转描述方式！q = w + xi + yj + zk，无万向锁问题
⚠️ 欧拉角 → 万向锁：当 pitch = 90°，第一次和第三次旋转等价，丢失一个表示维度。因此 ROS 使用四元数。
图8-5（附录）：万向锁 (Gimbal Lock) 现象示意图 — 欧拉角的致命缺陷
📖 图片说明：欧拉角使用 Roll（绕X轴翻滚）、Pitch（绕Y轴俯仰）、Yaw（绕Z轴航偏）三个角度描述旋转，直观易懂。但存在万向锁问题：当 Pitch = 90° 时，第一次旋转（Roll）和第三次旋转（Yaw）的转轴重合，导致丢失一个自由度（原本3自由度变成2自由度）。这就像图中两个环重叠在一起，无论怎么转都只能产生两个方向的旋转。因此 ROS 统一使用四元数 (Quaternion) 来描述旋转——四元数由 w + xi + yj + zk 组成，无万向锁问题。

第九章 SLAM 建图

Simultaneous Localization and Mapping

9.1 SLAM 简介 核心概念

  • SLAM = 同步定位与建图（Simultaneous Localization and Mapping）
  • 机器人在未知环境中逐步建立地图，同时确定自身位置
  • ROS 中地图是灰度图像（通常 .pgm 格式）
  • 黑色 = 障碍物，白色 = 可行区域，灰色 = 未探索
  • 地图 Topic：/map，类型：nav_msgs/OccupancyGrid
💡 锁存 (Latched)：地图采用锁存方式发布 — 无更新时维持上次内容，新订阅者自动收到最后发布的地图。减少通信带宽占用。
图9-1：ROS 中的地图 — 灰度图像（pgm 格式），黑色=障碍物，白色=可行区域，灰色=未探索
📖 图片说明：ROS 中的地图本质上是一张灰度图像（通常为 pgm 格式）。黑色像素表示障碍物（墙体、家具等），白色像素表示可通行区域，灰色像素表示尚未探索的未知区域。地图以 Topic /map 的形式维护，消息类型为 nav_msgs/OccupancyGrid。地图采用锁存 (Latched) 方式发布——没有更新时维持上次内容不变，新订阅者自动收到最后发布的地图，减少带宽占用。SLAM 建图过程就像拼图一样，在 RViz 中可以看到地图被逐步建立。

9.2 Gmapping SLAM 最重要

  • 基于粒子滤波 (RBPF)，最可靠和成熟的激光 SLAM 算法
  • 位于 slam_gmapping metapackage 中
  • 输入：/tf（base_frame↔laser_frame 和 base_frame↔odom_frame）+ /scan（激光雷达数据）
  • 输出：/tf（map_frame↔odom_frame 变换 = 定位）、/map（地图）、/map_metadata
  • 提供服务：/dynamic_map（nav_msgs/GetMap 类型）
⚠️ 里程计误差修正关键理解：Gmapping 不修正 odom_frame↔base_frame 的 tf！而是把误差补偿在 map_frame↔odom_frame 之间的 tf 上。里程计数据是多少就显示多少。
图9-2：Gmapping SLAM 计算图 — slam_gmapping 节点的输入输出关系
📖 图片说明：这是 Gmapping 算法的计算图（消息流向图）。中心的 slam_gmapping 节点负责整个 SLAM 工作。输入：① /tf — 必须包含 base_frame↔laser_frame（底盘与激光雷达的变换）和 base_frame↔odom_frame（底盘与里程计原点的变换）；② /scan — 激光雷达数据 (sensor_msgs/LaserScan)。输出：① /tf — 输出 map_frame↔odom_frame 之间的变换（这就是定位修正——把里程计误差补偿在地图和里程计原点之间）；② /map — 建立的地图；③ /map_metadata — 地图元信息。

9.3 Karto SLAM

  • 与 Gmapping 工作方式非常类似
  • 输入同样需要 /tf 和 /scan
  • 输出比 Gmapping 少了一个位姿估计分散程度
  • 同样提供 /dynamic_map 服务

9.4 Hector SLAM

  • 只需要激光雷达数据，不需要里程计！
  • 适合手持式激光雷达、没有里程计的机器人
  • 扫描频率有一定要求
  • 效果不如 Gmapping 和 Karto（仅用激光雷达信息）
  • 额外订阅 /syscommand Topic（接收 "reset" 消息时初始化）
  • 额外输出 /poseupdate 和 /slam_out_pose（位姿估计）

9.5 三种 SLAM 算法对比

算法传感器需求特点适用场景
Gmapping激光雷达 + 里程计粒子滤波 RBPF，最成熟可靠大多数轮式机器人
Karto激光雷达 + 里程计与 Gmapping 类似大多数轮式机器人
Hector仅激光雷达无需里程计，精度较低手持设备、无里程计机器人
图9-3：Karto SLAM 计算图 — 与 Gmapping 非常类似但输出略有不同
📖 图片说明：Karto SLAM 在工作方式上与 Gmapping 高度相似。输入同样是 /tf（需连通 odom_frame↔base_frame 和 laser_frame）和 /scan。唯一的区别在输出：slam_karto 的输出比 slam_gmapping 少了一个位姿估计分散程度 (/slam_gmapping/entropy)。两者都提供 /dynamic_map 服务（类型为 nav_msgs/GetMap）用于获取当前地图。

10.3 move_base 核心节点 重点

  • 包含多种插件：global_planner、local_planner、global_costmap、local_costmap、recovery_behaviors

⑤ move_base_simple/goal — 目标点

🔵 move_base 输出：
⑥ /cmd_vel — 规划的速度指令 (geometry_msgs/Twist)
